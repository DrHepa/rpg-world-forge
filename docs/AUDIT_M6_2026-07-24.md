# M6 Partial Implementation and Readiness Evidence — 2026-07-24

## Verdict

**PARTIAL — local implementation evidence only. Self-contained Studio release
remains blocked; hosted and native evidence is pending until the final push.**

M6 has delivered an interactive local authoring control plane, machine-readable
runtime-composition contracts, a static adapter seam, immutable composed
bundles, and representation-neutral standalone consumption. It has not delivered
a complete 3D game runtime or a redistributable self-contained Studio package.
This is M6 evidence and does not revise or reopen the completed
[M5 audit](AUDIT_M5_2026-07-21.md).

## Evidence method and limits

This record consolidates the committed local implementation and focused local
verification. No provider, model, paid service, Blender/Modly MCP backend,
runtime download, signing service, or graphical game session was invoked to
produce it. Exact hosted Ubuntu 24.04, Windows Server 2022, Linux x86_64 native,
and Windows-native results do not exist for the unpublished tree and remain
pending.

The runtime provenance contract is authoritative:
`apps/studio/packaging/runtime-sources.json` records
`release_ready=false` and redistribution status `blocked`. Therefore this
repository does not define self-contained artifact publication, signing, or
runtime-download CI. Shell-only package verification is evidence about the
Electron boundary only; it is not release evidence.

## Representation boundary

M6 can describe and validate 2D, 2.5D, 3D, 2D-over-2.5D, 2D-over-3D, and
2.5D-over-3D compositions while keeping UI and audio as orthogonal planes.
That composition work consumes M5's engine-neutral 3D asset handoff; it does
not turn 3D authoring evidence into a playable renderer.

The supported native path remains the existing Linux x86_64 2.5D adapter.
The bounded pyray proof demonstrates GLB animation mechanics only. No current
adapter proves complete 3D assetpack consumption, collision, physics,
navigation, representative performance, cross-platform native execution, or a
full game-release package. Those missing capabilities keep M6 partial.

## Self-contained Studio distribution boundary

All acquisition caches, assembled runtime resources, shell package directories,
ZIPs, games, and end-to-end artifacts must be created outside this repository.
Checked-in files contain provenance, schemas, source, and tests—not downloaded
Codex/Python payloads or generated packages.

The fail-closed runtime source contract lists these seven exact open blocker
codes:

1. `codex_ripgrep_static_dependency_notice_sbom_incomplete`
2. `linux_bwrap_lgpl_corresponding_source_relink_build_materials_incomplete`
3. `linux_bwrap_musl_provenance_incomplete`
4. `pbs_zlib_ng_license_incomplete`
5. `linux_berkeley_db_dbm_route_unresolved`
6. `windows_vc_runtime_redistribution_authority_unresolved`
7. `github_attestation_trust_root_rfc3161_verification_pending`

Technical acquisition success cannot waive these legal and provenance
requirements. Real assembly rejects the source contract before opening an
archive or creating an output. The synthetic assembly path proves deterministic
mechanics without redistributing the blocked payloads. Verified `shell_only`
packages retain all seven blockers, state `release_ready=false`, and prove that
Python and Codex runtimes are absent.

The accepted policy is recorded in
[ADR-0021](decisions/0021-studio-runtime-distribution-boundary.md).
