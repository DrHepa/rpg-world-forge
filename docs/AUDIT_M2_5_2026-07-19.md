# M2.5 implementation audit — 2026-07-19

## Verdict

M2.5 is complete as the presentation and asset-runtime foundation. A generated
game can now turn a canon-locked worldpack plus reviewed production evidence
into a small, provider-neutral renderpack and load it through pyray/raylib 6.
The game still runs without a renderpack by using primitive fallbacks.

This milestone deliberately does **not** generate assets. Automatic inventory,
OpenAI production adapters, Modly extension orchestration, deterministic
processing, and batch QA remain M5. M2.5 defines and enforces the contracts that
those later systems must produce.

## Resolved findings

| ID | Severity | Finding | Resolution |
| --- | --- | --- | --- |
| M25-01 | High | Asset planning trusted a declared worldpack hash without recomputing it. | Canon hashes are recomputed before planning, P10, release validation, and renderpack compilation. |
| M25-02 | High | The production manifest contained prompts, model/workflow data, licenses, and approvals that must never reach runtime. | `build-renderpack` emits a separate hashed artifact containing only copied processed files, semantic bindings, and world identity. |
| M25-03 | High | “Local model” was an unrestricted provenance label. | Local generation is accepted only through a named/versioned Modly extension and checked-in workflow. Modly is opt-in; OpenAI is the default route. |
| M25-04 | High | Runtime asset paths and files had no independent containment/integrity boundary. | The loader confines paths below the renderpack root, applies size limits, verifies every SHA-256, and rejects a mismatched world ID/hash. |
| M25-05 | Medium | Media declarations could disagree with the output role or underlying bytes. | Role/media compatibility, signatures, PNG dimensions, WAV properties, hashes, and per-specification budgets are validated before packaging. |
| M25-06 | Medium | Spritesheet layout and timing were implicit. | Versioned clipsets now define finite rectangles, pivots, loops, and integer tick durations; bounds are checked against the loaded texture. |
| M25-07 | Medium | Asset paths were not connected to world semantics. | Typed bindings cover actors, tile types, interactions, portraits, events, map/default music, UI, abilities, and scenes. Release requires every actor and tile type. |
| M25-08 | Medium | Malformed JSON types could reach Python set/dict operations and raise `TypeError`. | Validators now return structured issues for invalid kinds, statuses, routes, roles, media types, bindings, and clips. |
| M25-09 | Medium | Raylib resource failure could leave a partially initialized presentation stack. | Handles are checked with raylib 6 validity APIs; partial loads are unwound in reverse order and cleanup remains best-effort before window shutdown. |
| M25-10 | Medium | P13 could complete with production evidence but no runnable asset bundle. | The phase gate now requires both a release-valid manifest and a renderpack that loads against the exact P10 worldpack. |

## Implemented runtime surface

- `isoworld --renderpack <path>` loads an optional `isoworld.renderpack`.
- Textures, fonts, shaders, SFX, and streamed music have explicit ownership and
  cleanup.
- Actors, terrain, and interactions use texture bindings with primitive
  fallbacks.
- Animations select deterministic frames from simulation ticks and can switch
  to a moving clip while an actor follows a route.
- Actors/interactions use stable isometric depth plus an authored layer.
- Dialogue speakers may have portraits; domain events may have specific or
  generic SFX; maps may override default music.
- `Camera2D` owns drawing and inverse click transforms, with bounded zoom and
  middle-button pan.
- No AI, OpenAI, Modly, prompt, model, extension, or network dependency exists
  in `src/isoworld`.

## Authoring-route policy

`worldforge init-assets` enables only `openai` by default. That route represents
GPT, Codex, and GPT Image used outside the game. A game that chooses local
models explicitly adds `modly` to `generation_policy.enabled_routes`; every
local candidate then records its Modly extension ID, extension version, model
revision, recipe, and workflow file.

Neither route executes inference in this milestone. The runtime receives only
approved processed files through the renderpack.

## Accepted limits before M3

- Direction/facing and action-specific animation selection wait for the related
  simulation state.
- Walls, props, occlusion, multilayer maps, culling, virtual resolution, and
  shader application policies remain future presentation work.
- Automatic slicing, atlasing, image cleanup, audio normalization, and asset
  generation are not implemented yet.
- A renderpack is an integrity-checked build artifact, not a signed trust
  package. Distribution signing can be added at the game/release layer.

These limits do not block M3 living-world simulation. Before declaring a public
release, the Xvfb raylib smoke job must pass in GitHub Actions.
